# ① 課題名

TriArk チャット — 陸海空を繋ぐ旅のプラットフォームの予約チャット

## ② 課題内容（どんな作品か）

- Firebase（Authentication + Firestore）で作った、ログイン認証付きのリアルタイムチャットアプリ
- 私が構想している旅行プラットフォーム「TriArk」の、ゲスト⇄ホスト間メッセージ機能を想定して制作
- 単なるチャットではなく、Airbnb のような「予約プラットフォームのチャット」の仕組みを再現することを目標にした

## ③ アプリのデプロイURL



## ④ アプリのログイン用IDまたはPassword

- メール／パスワードで新規登録すれば、どなたでも利用可能
- 動作確認用の例 — ゲスト: guest@test.com / host@test.com、PW: 123456
- 登録時に「ゲスト」「ホスト」の役割を選択できる

## ⑤ 工夫した点・こだわった点

- 役割（ゲスト／ホスト）を登録時に分け、それに応じて表示や操作を出し分けた
- 予約ステータス（問い合わせ中 → 承認待ち → 予約確定）を可視化し、ゲストがリクエスト・ホストが承認する2段階フローを実装
- ステータスが変わると「予約をリクエストしました」等のシステムメッセージが自動で会話に挿入される（Airbnb の体験を再現）
- 既読表示を、各メッセージではなく「相手の最終既読時刻」をルームに記録する軽い方式で実装
- 名前の頭文字を丸いアイコンにし、役割で色分け（ゲスト=シアン、ホスト=オレンジ）
- ブランドロゴ（陸・海・空の三つの弧と生命の樹のT）と、それに合わせた配色でデザインを統一
- Firestore のクォータを無駄に消費しないよう、既読更新の頻度を制限するなど書き込み回数に配慮

## ⑥ 難しかった点・次回トライしたいこと

- 既読更新の処理が無限ループになり、Firestore のクォータを使い切ってしまった。原因を特定して書き込み制御を入れて解決した
- AI翻訳機能をブラウザから直接 APIに繋ごうとしたが、セキュリティ制限（CORS）で動かなかった。
- 次回は、サーバー（バックエンド）を学んで AI翻訳（日本語↔英語）を本実装したい。TriArk は国境を越えたコミュニティを目指すので、翻訳は中核機能になる
- ゲスト⇄ホストのペアごとに個別ルームを作る仕組みにも挑戦したい（今回は共通ルーム）

## ⑦ フリー項目（感想など）

- 「ただ動くチャット」ではなく「予約サービスとして成立するチャット」を、事業の仕組みから逆算して設計できたのが楽しかった。
-  普段ホストとしてゲストとチャットしているデータの仕組みを垣間見たことは非常に感慨深いものだった。



## 6/19までの課題「APIをつかったアプリ」

# ① 課題名

TriArk チャット — 天気・気象警報APIを組み込んだ、旅の安全判断つき予約チャット

## ② 課題内容（どんな作品か）

- 前回制作した TriArk チャット（Firebase 認証＋リアルタイムチャット）に、複数の外部APIを組み込んで発展させた
- Open-Meteo（天気API）で、ホストの活動エリア（仮：浦安マリーナ）の気温・天気・風速を自動表示
- 気象庁の防災情報JSONで、同エリアの警報・注意報をリアルタイムに取得し、催行（出航）可否の判断材料として表示
- ゲスト・ホストが検索する手間なく、チャットを開くだけで「今の天気」と「公式の安全情報」が目に入る設計

## ③ アプリのデプロイURL

https://（あなたのGitHubユーザー名）.github.io/TriArk_chat/

（APIキーを伏せているため、公開ページは見た目のみ。動作はローカル環境で確認）

## ④ アプリのログイン用IDまたはPassword

- メール／パスワードで新規登録すれば利用可能（登録時にゲスト／ホストを選択）
- 動作確認用の例 — guest@test.com / host@test.com、PW: 123456

## ⑤ 工夫した点・こだわった点

- 単なる天気表示ではなく、「天候に左右されるサービスをどう設計するか」という固有の課題に踏み込んだ
- 細かい天気（降水確率など）はキャンセル判断に使いにくいと考え、気象庁の警報・注意報という"公式の客観基準"を催行判断の材料にした
- 海の体験（マリーナ）を想定し、風速や波浪・濃霧など、船の出航判断に直結する情報を重視した
- 「気象庁の警報をそのまま表示し、可否はホストが判断」という形にし、独自の予報を断定的に提供しない設計にした
- 地名検索をさせず、エリアの座標・コードをあらかじめ持たせることで、ゲスト・ホスト双方の入力の手間をゼロにした
- 利用したデータの出典（Open-Meteo／気象庁）を画面下に明記し、規約を順守した

## ⑥ 難しかった点・次回トライしたいこと
- 気象庁のJSONは「正式なAPI」ではなく、データの在り処や構造をAIで調べる必要があった
- 気象業務法上、気象庁以外が「警報」を発表することはできず、独自の予報を繰り返し提供するには許可が要ると知った。今回「公式情報をそのまま見せ、判断は人に委ねる」設計にしたのは、この理解による
- 次回は、予約日と連動した「その日の天気・警報」の表示や、体験タイプ（海／陸）による注目すべき警報の出し分けに挑戦したい
- 商用化する際は、商用利用が許可された有料の気象APIへの切り替えが必要、という前提も理解した

## ⑦ フリー項目（感想など）

- 今回の一番の学びは、技術そのものより「天候という制御できないものを、サービスとしてどう扱うか」の難しさと、それが差別化にもなりうるという気づきだった。完璧な予測は不可能だからこそ、公式情報に判断を委ねる誠実な設計が信頼につながると感じた。
- APIには「キー不要で使えるもの（Open-Meteo・気象庁）」と「キーが必要なもの」があり、利用前に"無料か／商用可か／出典は要るか"を確認する大切さを学んだ。
